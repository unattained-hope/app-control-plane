An internal control plane to administer your Shopify app portfolio

A concrete, opinionated reference architecture for one admin app that manages SaleSwitch today and many apps tomorrow — merchant management, an in-house support inbox, subscriptions, and per-app analytics — built on a shared-DB read model with a per-app plugin/connector seam.

00 TL;DR & recommended reference stack

If you read nothing else: build a persistent (non-serverless) TypeScript app, read each app's Postgres through a per-app read-replica connector, and base the support inbox on Chatwoot rather than inventing one.

★ The one-paragraph verdict

Build the control plane as a persistent-server, full-stack TypeScript app — React Router 7 + tRPC + Prisma + Postgres (the T3 composition, but on the router your team already knows) — with a per-app connector layer that holds one long-lived Prisma client per app database, each pointed at a read-replica via Prisma's official read-replicas extension 3,4. Wrap the merchant/subscription/analytics surfaces in a plugin/module registry so each new app "plugs in" its data source + screens. For support, base the inbox on Chatwoot (MIT) rather than building from zero 5, and ship a thin embeddable widget that respects Shopify's per-shop CSP frame-ancestors rules 6. Authenticate the internal team with a cookie role login behind a zero-trust gateway / Basic Auth. Do not deploy this to Vercel/serverless — the WebSocket inbox, multi-DB connection pools, and BullMQ workers all want a long-running process.

01 Context & constraints

The four stakeholder decisions this report is built around.

Purpose
One app, many apps

A single internal admin to administer the whole Shopify-app portfolio. SaleSwitch is tenant #1; the top-bar app-selector switches context, and each app may expose different modules, analytics, and pages. Build for one, design for many.

Decision · Stack
Best-in-class, open

Pick the strongest stack for an internal control plane — not constrained to current choices, but free to reuse the team's React Router 7 / Node / Prisma / Postgres / BullMQ muscle memory where it's genuinely best.

Decision · Data
Shared DB / direct read

The admin app reads each app's Postgres directly (shared or replica). Must stay safe as apps multiply — bounded connections, least-privilege, and a clean exit toward decoupling later.

Decision · Support
Build in-house

An embeddable chat widget inside each Shopify app, with conversations surfacing in a support inbox in the admin app. Owns its data and branding.

◆ A reframing worth internalizing ✓8

Microsoft's multitenancy guidance is blunt: "A control plane functions as an application. You must design your control plane with the same rigor and care that you apply to other parts of your solution." This is not a side dashboard — it is a product, with its own SLA, auth, audit, and tests. It also warns that a manual/spreadsheet catalog only works below ~10 tenants, with the automate-now threshold "typically between 5 and 10." You're at 1 app, so a lightweight registry is fine today — but bake in the seam now so app #5 doesn't force a rewrite.

02 Architecture overview

A control-plane core, a per-app connector/plugin seam, and an in-house chat subsystem — all reading from app read-replicas, never their primaries.

Control-plane core Connector / plugin seam Realtime + read path Shopify integration

The whole design hinges on one seam: the per-app connector. Each app contributes (a) a read-only Prisma client to its replica, and (b) a set of UI modules + analytics queries. The control-plane core is app-agnostic; the top-bar selector simply swaps which connector is active. This is the practical implementation of the "tenant catalog + deployment stamps" idea from control-plane theory 8.

05 Shared-DB connector design

How to read each app's Postgres directly today, safely, and keep an exit ramp toward decoupling when coupling starts to hurt.

The pattern: one client per app DB, each on a read-replica

Prisma's officially documented approach for multiple databases is a separate schema + a separate long-lived PrismaClient per database — explicitly endorsed "for multi-tenant applications or when you need to separate concerns when managing connections to multiple databases." 3 A schema allows only one datasource block, so a single multi-datasource client is not an option. That maps exactly onto a per-app connector.

◆ Connector contract (sketch)

Each app registers an object implementing a shared interface — the seam that makes the control plane app-agnostic:

interface AppConnector {
  key: "saleswitch" | "app2" | ...;       // tenant id in the registry
  db: PrismaClient;                         // long-lived singleton, READ-REPLICA url, read-only role
  listMerchants(q): Promise<MerchantRow[]>;  // app maps its own schema → common shape
  getMerchant(shop): Promise<MerchantDetail>;
  subscriptions(shop): Promise<SubState>;     // see §8 — reads Shopify-owned state
  modules: AdminModule[];                    // app-specific screens/analytics (plugin)
}

Read-replica routing — official, automatic

Point each connector's client at a read-replica using Prisma's official @prisma/extension-read-replicas. It auto-routes all reads (findMany, etc.) to replicas and sends all writes + $transaction to the primary, with optional $primary()/$replica() overrides — no manual per-query routing. 4 This keeps the control plane's read load entirely off each app's production primary.

RuleWhyConfidence
One long-lived client per app DB (singleton)Prisma-endorsed multi-tenant pattern; avoids the per-request "new client" pool-exhaustion anti-pattern.✓3
Target a read-replica via the official extensionReads auto-route to replica; writes/transactions to primary. Control plane never writes app DBs.✓4
Dedicated read-only Postgres role per appLeast privilege — the admin credential physically cannot mutate or drop merchant data.◆
Keep client count bounded; cap pool size per clientEach client holds memory + its own pool. Fine at N apps; the trigger to decouple is when N grows large.✓3
Tolerate replication lag in the UIRead-after-write lag means the admin sees slightly stale data — acceptable for an admin read model; surface "as of" timestamps.✓4
⚠ Security: never decrypt app tokens in the control plane

SaleSwitch stores access tokens AES-256-encrypted at rest. The admin plane should not read or decrypt them. Two concrete hazards from the evidence: (1) Prisma $queryRaw/$executeRaw default to the PRIMARY DB (the extension can't infer read-vs-write intent for raw SQL) — so a raw query to "just peek" at an encrypted column bypasses your replica isolation. 4 (2) Decryption keys must never be co-located with the read-only admin role. If a workflow genuinely needs a decrypted value, do it through a narrow, audited API the owning app exposes — not by reaching into its columns.

Schema-coupling risk & the exit ramp

Direct shared-DB reads are the right call for one app and a small portfolio. The known downside is coupling: an app's internal schema change can silently break the admin plane. Mitigations and the migration trigger:

  • Version the connector, not the schema. Each connector maps the app's raw tables → a stable common shape (MerchantRow, SubState). When an app refactors, only its connector changes — the core is insulated.
  • Contract tests per connector. A nightly test that runs each connector's queries against the replica catches a breaking schema change before a human does.
  • The decouple trigger: when client count / coupling pain crosses the same ~5–10-tenant line where manual catalogs break down 8, evolve specific apps from direct reads to a thin read-API or CDC/event stream (Debezium → Kafka/Redpanda, or Postgres logical replication into a control-plane store). The connector interface stays identical — only its implementation swaps from "Prisma to replica" to "call app API." That's the whole point of the seam.

06 In-house support chat & inbox

"Build in-house" — but stand on Chatwoot rather than reinventing conversation management, and respect Shopify's embedded-iframe rules on the widget side.

★ Don't build the inbox from zero — base it on Chatwoot (MIT)

Chatwoot is an MIT-licensed OSS live-chat / omni-channel helpdesk ("an alternative to Intercom, Zendesk"). Its community edition already ships the exact features the brief calls for: auto-assignment/routing, teams, private notes & @mentions, labels, canned responses, and CSAT. 5 Either self-host the MIT core and embed its agent inbox, or treat its data model as the reference design for a custom inbox. Building auto-routing + threading + presence from scratch is months you don't need to spend.

Open-core caveat: since v2.0 Chatwoot is open-core — the community edition is MIT, but SSO/SAML, audit logs, SLA, and white-labeling live under a separate proprietary enterprise license. For an internal tool the MIT core is plenty; just don't assume those specific enterprise features are free. 5

The hard part is the embedded widget, not the inbox

The widget ships inside each Shopify app's embedded admin (App Bridge, in an iframe under admin.shopify.com). That imposes constraints the research verified:

07 Multi-app control-plane patterns

The app-selector, the plugin/module system, and the registry — designed so app #5 is a config change, not a refactor.

  • App registry (tenant catalog). A first-class table of apps: key, display name, theme tokens, connector config, replica DSN ref, enabled modules, feature flags. This is the "tenant catalog" from control-plane theory — config + which "deployment stamp" (DB) each app maps to. 8
  • Module/plugin system. Each app contributes AdminModule[] — a route, a nav entry, an RBAC scope, and its analytics queries. The top-bar selector filters the nav to the active app's modules. New module = one file + a registry entry, mirroring how SaleSwitch already does wizard commands.
  • Per-app theming. Theme tokens (logo, accent) live in the registry; shadcn/ui + CSS variables re-skin per active app so the inbox/dashboards feel native to each brand.
  • Feature flags / remote config. Per-app, per-merchant, and per-internal-role flags to dark-launch admin modules and gate destructive actions. (Unleash self-hosted, or a simple Postgres-backed flag table to start.)
  • Lifecycle processes. Onboarding a new app = register connector + replica + modules; offboarding = revoke role + archive. Keep these as scripted, reviewed processes, not tribal knowledge — the same rigor bar as the apps themselves. 8

08 Shopify-specific: billing & mandatory GDPR

Two non-negotiables: read subscription state from Shopify (don't build a billing ledger), and make sure every app ships the three compliance webhooks.

◆ Billing: read Shopify-owned state, don't own a ledger ✓9

App-Store apps are required to use a Shopify-provided billing solution; off-platform billing is effectively prohibited. The canonical mutation to charge is appSubscriptionCreate (recurring/usage AppSubscription). For the control plane, the load-bearing operation is reading: query currentAppInstallation.activeSubscriptions (Admin API) per shop — or the Partner API activeSubscription under Shopify App Pricing. Net: the admin app surfaces plan/MRR/trial/cancellation state read from Shopify; it does not run its own billing engine. (Note: Shopify now frames "Shopify App Pricing"/Managed Pricing as the default and the manual Billing-API path as "legacy" — not deprecated; this mainly changes which query returns state.)

⚠ GDPR compliance webhooks are a hard App-Store gate ✓10

Every app distributed through the App Store must implement three mandatory compliance webhook topics — customers/data_request, customers/redact, shop/redact — regardless of whether it collects personal data. Each must: verify the Shopify HMAC header (return 200-series on valid, 401 Unauthorized on invalid), complete the requested action within 30 days, and be reachable — missing/invalid URLs cause App-Store rejection. This is a per-app obligation the control plane is well placed to monitor: a compliance module tracking webhook delivery health, redaction-request SLAs, and merchant lifecycle state per app/merchant is a concrete high-value feature beyond the core MVP.

Merchant / shop lifecycle

Track install / uninstall (app/uninstalled), trial → paid → churned, and shop status (active/frozen/closed) per merchant. Most of this is derivable from the shared-DB read model plus Shopify subscription state; surface it as the merchant "health" timeline (§9, §12).

09 Audit logging & admin-action safety

An internal tool with write powers over merchants is a high-value target and a foot-gun. Treat audit + guards as MVP, not later. ◆

  • Append-only audit log of every admin action: who, what, which app/merchant, before/after, when, and the request context. This is also your GDPR/SOC-readiness evidence. Build it in your own store — don't assume a framework gives it to you (Refine doesn't 2).
  • Destructive-action guards. Type-to-confirm modals for irreversible actions; two-person approval for the most dangerous (e.g. force-cancel a subscription, bulk merchant ops). Mirror SaleSwitch's existing WarningModal discipline.
  • Impersonation / "view as merchant" must be explicit, time-boxed, banner-flagged, and audited — impersonation is a well-known privilege-escalation and accountability risk; never let an impersonated session perform writes silently. 11
  • Least privilege end-to-end: read-only DB roles per app (§5), RBAC scoping per module, and SSO-gated access (§11). The admin credential should never be able to corrupt an app's production data.

15 Risks, caveats & open questions

⚠ Time-sensitive / version risks to check before you build
  • Prisma read-replicas extension — the current package reportedly requires Prisma 7.0+ while your apps are on Prisma 6. Routing semantics are stable across versions, but verify compatibility or plan the upgrade before adopting. 4
  • Auth.js in maintenance mode since Sept 2025 — fine as a free fallback, not the default for a new tool. 1
  • Shopify "App Pricing" now the default; appSubscriptionCreate reframed as "legacy" (not deprecated) — mainly affects which query returns subscription state. 9
  • Chatwoot open-core — SSO/SAML, audit logs, SLA, white-labeling are not MIT. Self-host the core; don't assume those are free. 5
✗ The one refuted assumption — honor it

"Refine ships built-in auth, RBAC, audit-log, and realtime providers as finished features" was refuted (1-2). Refine gives you provider interfaces you must wire to a backend — not turnkey RBAC/audit/chat. Don't let a framework choice rest on that belief. 2

Open questions deferred to a dedicated evaluation (no comparative evidence survived verification)

  • UI component library — shadcn/ui vs Mantine vs AntD vs MUI: recommended on judgment (shadcn/ui), worth a half-day spike against your real merchant grid before committing.
  • Realtime transport — Socket.IO recommended on judgment; benchmark against an embedded-iframe cross-origin handshake early.
  • Hosting platform — pick driven by where your app read-replicas physically live (co-location beats cleverness).
  • Decoupling trigger & connector versioning — define the concrete schema-change contract test and the tenant-count threshold that flips an app from direct reads to API/CDC.
